| وصف البنود + الكمية + السعر | ✅ موجود | `items[]` |

| الإجمالي شامل الضريبة | ✅ موجود | |
| رمز QR (للمبسّطة) | ✅ موجود | `qr_code` field |
| نص عربي | ✅ موجود | كل الواجهة عربية |
| عدم تعديل الفواتير المُصدرة | ⚠️ جزئي | يحتاج تأكيد عبر RLS policy |

| المتطلب | الحالة | الملاحظات |

| Status tracking | ✅ موجود | `zatca_status` (draft/cleared/reported/rejected) |

## 🛠️ الإجراءات المطلوبة قبل التواصل مع ZATCA

### 1️⃣ تسجيل النظام في ZATCA كـ "حل فوترة معتمد" (Software Solution)

- [ ] التسجيل في **بوابة Fatoora**: https://fatoora.zatca.gov.sa
- [ ] الحصول على **EGS Onboarding Token**
- [ ] إعداد بيئة الـ Sandbox لاختبار التكامل
- [ ] الحصول على **Compliance Certificate** من ZATCA Test environment
- [ ] تشغيل **اختبارات الامتثال** على 200+ سيناريو فاتورة (XML invoice samples)

### 2️⃣ تنفيذ مكوّنات المرحلة الثانية الناقصة

#### A. توليد XML الموافق لـ UBL 2.1
```
الحالة: ❌ غير منفّذ
المطلوب:
- توليد XML بصيغة UBL 2.1 لكل فاتورة
- يشمل جميع الحقول الإلزامية (BR-CO-26 و BG-13 و BG-14...)
- توقيع الـ XML بالـ ECDSA private key
```

#### B. الختم التشفيري (Cryptographic Stamp)
```
الحالة: ⚠️ جزئي
المطلوب:
- استدعاء ZATCA Compliance CSID API
- استخدام مفتاح ECDSA لتوقيع كل فاتورة
- تضمين التوقيع في XML
```

#### C. سلسلة الـ Hash (PIH)
```
الحالة: ⚠️ جزئي
المطلوب:
- حساب SHA-256 لكل فاتورة
- ربط كل فاتورة بـ hash السابقة (chain)
- المخالفة = رفض من ZATCA
```

#### D. التكامل مع Fatoora API
```
الحالة: ❌ غير منفّذ
المطلوب:
- Standard invoices: POST /clearance/single → ينتظر موافقة قبل إرسالها للعميل
- Simplified invoices: POST /reporting/single → خلال 24 ساعة
- معالجة الردود: ACCEPTED / ACCEPTED_WITH_WARNINGS / REJECTED
```

#### E. QR Code الكامل (TLV format)
```
الحالة: ⚠️ يحتاج تأكيد
المطلوب: QR يحتوي 5 حقول إلزامية بصيغة TLV (Tag-Length-Value):
1. اسم البائع
2. الرقم الضريبي
3. التاريخ والوقت
4. إجمالي الفاتورة
5. إجمالي الضريبة
6. (Phase 2) Hash الفاتورة
7. (Phase 2) التوقيع التشفيري
8. (Phase 2) Public Key
9. (Phase 2) ECDSA توقيع التشفير
```

### 3️⃣ ميزات إدارية مطلوبة

| الميزة | الحالة |
|--------|--------|
| سجل تدقيق (Audit Log) | ✅ عبر `journal_entries` |
| الاحتفاظ بالبيانات 6 سنوات+ | ✅ Supabase Pro (نسخ احتياطية) |
| تشفير في الراحة (encryption at rest) | ✅ Supabase افتراضياً |
| تشفير أثناء النقل (TLS) | ✅ HTTPS + TLS 1.2+ |
| منع التلاعب في الفواتير | ⚠️ يحتاج تأكيد |
| إصدار ملاحظات الدائن (Credit Notes) | ❌ غير موجود |
| إصدار ملاحظات المدين (Debit Notes) | ❌ غير موجود |
| دعم العملات الأجنبية | ❌ غير موجود (ر.س فقط) |

## 🎯 خطة العمل قبل التقديم لـ ZATCA

### المرحلة A — اعتماد المرحلة الأولى (سريع)
**المدة المتوقعة: 1-2 أسبوع**

1. ✅ التحقق من جميع حقول الفاتورة الإلزامية
2. ⚠️ التحقق من QR code يحتوي الحقول الـ 5 الأساسية بصيغة TLV
3. ❌ تطبيق Credit Notes + Debit Notes
4. ⚠️ تأكيد أن الفواتير المُصدرة غير قابلة للتعديل (RLS policy + journal entries)
5. 📝 إنشاء **شهادة الامتثال للمرحلة 1** (Phase 1 Compliance Declaration)

**هذا كافٍ لـ:** بدء بيع النظام للعملاء كحل لـ Phase 1 (90% من المنشآت السعودية حالياً)

### المرحلة B — اعتماد المرحلة الثانية (موسّع)
**المدة المتوقعة: 4-8 أسابيع**

1. ❌ تطوير محرّك XML UBL 2.1
2. ❌ تطبيق ECDSA signing
3. ❌ تطبيق PIH chain
4. ❌ تطبيق Fatoora API integration (sandbox)
5. ❌ تشغيل 200+ سيناريو اختبار
6. ❌ الحصول على Compliance Certificate من Sandbox
7. ❌ الحصول على Production Certificate
8. 📝 إعلان امتثال المرحلة 2

**هذا كافٍ لـ:** بيع للمنشآت الكبيرة المُلزمة بالـ Phase 2

## ✅ التوصية النهائية

**قبل ما تتواصلين مع ZATCA:**

1. **إذا تبين الاعتماد للمرحلة 1 فقط (سريع، 95% جاهز):**
   - أضيفي Credit/Debit Notes
   - تأكدي من QR TLV format
   - اطلبي **Compliance Declaration** فقط

2. **إذا تبين الاعتماد للمرحلة 2 (موسّع، 50% جاهز):**
   - يحتاج 4-8 أسابيع تطوير إضافي
   - أو يمكن **التعاون مع مزوّد ZATCA معتمد** (مثل: Fatoora Plus, Wafeq) للاستفادة من تكاملهم الموجود

3. **الطريق الذهبي للسرعة:**
   - ابدأي بالمرحلة 1 **الآن**
   - أعلني المرحلة 1 جاهزة وابدأي البيع
   - طوّري المرحلة 2 بالتوازي مع نمو العملاء
